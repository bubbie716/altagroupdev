Alta Bank — Entity & Data Model Audit
========================================
Scope: Core data model only (altaweb/prisma/schema.prisma + entity usage
patterns). No UI, customer workflows, or code quality review.

Perspective: Chief Systems Architect — final architecture review before
long-term development.

Method: Full schema review (58 models, ~90 enums), cross-referenced with
Commercial).

Date: July 3, 2026


================================================================================
1. OVERALL ARCHITECTURE SCORE
================================================================================

61 / 100 — Strong Banking Core, Incomplete Capital Markets Foundation

Domain                              Score   Assessment
----------------------------------  ------  ------------------------------------------
Identity & access (User, tags)      78      Clean, extensible
Company & membership                80      Well-normalized junction model
Core banking (accounts, txns)       72      Functional but overloaded; denormalized balance
Payments (Alta Pay, scheduled)      65      Works; Alta Pay is not a first-class entity
Lending (applications, loans)       75      Mature sub-ledger pattern
Alta Card                           74      Complete card lifecycle; separate ledger
Deal rooms & secure threads         58      Dual architectures; triplicated thread models
Ops & audit (AuditLog, notes)       70      Good intent; weak polymorphic FK integrity
Relationship intelligence           55      Over-designed CRM mirror; premature denorm
Alta Private / Commercial           50      Tag + invitation only; no relationship entity

Verdict: Alta's banking schema can support current products without redesign.
major new schema layers. Building those products on today's model without first
introducing shared abstractions will produce fragmentation and painful
migrations.

9. NORMALIZATION RECOMMENDATIONS
================================================================================

Merge
-----
- Three secure thread types → one SecureThread + SecureThreadMessage
- User and Company relationship CRM stacks → parameterized by subject type
- Deal room messaging (legacy + V1) → single conversation abstraction

Split
-----
- BankTransaction → separate PaymentInstruction (pending request) from
  LedgerEntry (posted movement), OR add explicit role: REQUEST | LEG | REVERSAL
  discriminator
- TransferContact → IntrabankContact and WireBeneficiary tables
- PayrollRun.lineItems → PayrollRunLine normalized table (partially exists in
  execution layer only)

Add junction tables
-------------------
- TransferGroup (id, type, status) linking 2+ BankTransactions
- AltaPayPayment linking payer OUT + payee IN transactions
- InstrumentListing (companyId, instrumentId, listingDate, status)
- PortfolioHolding (accountId, instrumentId, quantity, costBasis)
- StaffClientAssignment (staffUserId, clientUserId/companyId, product, assignedAt)

Add lookup tables
-----------------
- FinancialInstitution (name, routingPrefix, status)
- InstrumentType (EQUITY, BOND, ETF, CRYPTO, INDEX)
- CorporateActionType lookup

Fix duplicate fields
--------------------
- LoanApplicationThread.assignedStaffId — remove or unify with
  DealRoom.assignedOfficerId
- Company.desiredTicker vs Company.ticker — correct pattern; ensure Exchange
  Listing references official ticker only
- Thread status enums duplicated across 3 thread models — single enum

Will survive at target scale

Bottlenecks

Bottleneck                          Cause
----------------------------------  ------------------------------------------
Denormalized balances               BankAccount.balance, AltaCard.currentBalance,
                                    Loan.outstandingBalance — reconciliation at scale is
                                    expensive
Polymorphic queries                 AuditLog, InternalNote, OpsReviewFlag filter by (type,
                                    id) without FK indexes on target
RelationshipProfile recalc          Aggregating all user/company financial data on each cron
                                    run — O(users × products)
JSON columns                        PayrollRun.lineItems, DealRoomAgreement.fieldData,
                                    notification metadata — hard to query at scale
No table partitioning strategy      Single BankTransaction table for all types will exceed
                                    10M rows
SecureDealRoomDiscordSession        One row per deal room; Discord channel proliferation
Duplicate CRM tables                8 relationship models × 2 subjects = maintenance burden
                                    doubles

Index gaps for scale
- BankTransaction: missing composite (bankAccountId, createdAt DESC) for statements
- AuditLog: missing (entityType, entityId, createdAt)
- UserNotification: missing (userId, createdAt DESC) for pagination — has (userId, readAt)

================================================================================
15. FINAL VERDICT
================================================================================

If Alta continues building on this data model, will it still be solid after
future financial products?

No — not without deliberate foundational work first.

What will hold up:
- User, Company, and membership model — extensible to issuers, brokerages,
  institutions
- Core banking account and transaction model — can serve as cash leg for
  settlement
- Loan and card sub-ledger patterns — good templates for other credit/instrument
  ledgers
- Audit, ops flags, and platform settings — cross-product infrastructure
- Payroll and scheduled payments — commercial banking foundation exists

What will break or require painful migration:
- Building Exchange/Terminal as separate mock-then-bolt-on schemas will
  duplicate Company/ticker concepts and diverge from banking
- Alta Pay and transfers as implicit transaction pairs will become unmaintainable
  at millions of transactions
- Three duplicate secure thread models will multiply with every new product
- Denormalized balances across bank, card, and loan without unified
  reconciliation will corrupt at concurrency scale
  no fulfillment path
- Relationship CRM duplication (user + company × 4 tables each) will not scale
  to 50k subjects

Recommended path:

1. Now: Introduce Instrument, Payment/TransferGroup, and FinancialInstitution as
   shared abstractions — even before Exchange UI goes live.
2. Before Exchange: Add Listing, BrokerageAccount, Holding — anchor to existing
   Company.ticker.
   interbank to reference Institution FK.
4. Consolidate: Merge secure threads and relationship CRM before adding
   Commercial/Private RM workflows.

Bottom line: Alta's banking schema is institution-grade for V1 bank + card +
lending. It is not yet a capital markets foundation. Treat the next schema sprint
as infrastructure, not feature work — add 5–8 generic entities before building
Exchange or Terminal features, or expect a costly redesign within 12–18 months.
