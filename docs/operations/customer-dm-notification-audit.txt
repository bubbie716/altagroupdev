Alta Customer Discord DM Notification Audit
=============================================
Scope: altaweb — all customer-facing Discord DMs across banking, commercial,
merchant, payments engine, deal rooms, credit, companies, and security.
Date: July 5, 2026
Status: Audit only — no code changes.

Note: docs/operations/customer-dm-notification-coverage-report.txt (July 3) is
partially outdated. Many items it lists as missing (transfer/Alta Pay failures,
scheduled transfers, loan/card payments, staff deal-room messages) are now
implemented.


================================================================================
1. OVERALL DM NOTIFICATION COVERAGE SCORE
================================================================================

Metric                                              Score
--------------------------------------------------  -----
Overall coverage (events that should exist today)   ~74%
Banking core (deposits, transfers, Alta Pay, etc.)  ~90%
Operator / security actions                         ~95%
Merchant invoices & payment links                   ~75%
Commercial Pro billing                              ~85%
AutoPay & payment engine                            ~60%
Deal rooms                                          ~70%
Credit (loan + Alta Card)                           ~65%
Companies & invitations                             ~55%

Delivery infrastructure: Strong. Three paths (standard, operator, invitation)
with retry queue (5 attempts: 5m, 15m, 1h, 4h, 24h), delivery audit, and
dedupe keys.

Systemic gaps dragging the score:
  - paymentEngineNotificationPrefs stored but never enforced
  - Merchant/commercial/deal-room types not exposed in user settings UI
    (always default-on)
  - Several enum types defined in Prisma but never sent
  - Invitation DMs have no retry queue (login re-sync only)
  - Raw/internal error strings can leak into payer-facing failure copy


DELIVERY ARCHITECTURE (REFERENCE)
---------------------------------

Path          Entry                                              In-app  Prefs   Retry  Silent
------------  -------------------------------------------------  ------  ------  -----  ------
Standard      createUserNotification -> deliverCustomer...       Yes     Yes     Yes    No
Operator      deliverOperatorCustomerNotification                No      Bypass  Yes    Yes*
Invitation    dispatchInvitationDm                               No      No      Re-sync No
Deal room     createUserNotification(skipDiscord) + deliver...   Yes     Yes     Yes    No

* Silent mode with restrictions (see section 9).

Standard embeds: title + body + optional "View on Alta Bank" link button
(lib/discord/notification-dm.ts).

9. SILENT MODE RECOMMENDATIONS
================================================================================

CURRENTLY IMPLEMENTED (silent-notification-restrictions.ts)
-----------------------------------------------------------

Forbidden (correct):
  - deny_deposit, deny_withdrawal
  - alta_pay_reversal, payment_reversal
  - account_freeze, account_restriction
  - company_verification_rejection, company_verification_revocation
  - loan_denial, alta_card_denial
  - Kinds: account_frozen, account_restricted, payment_blocked, payment_reversed

Allowed:
  - manual credit/debit, fees, corrections
  - holds (non-restriction), unfrozen, close/reopen
  - transaction review

Silent rejection creates OPS_SILENT_NOTIFICATION_REJECTED audit — good.


RECOMMENDATIONS
---------------

Action                          Recommendation
------------------------------  ------------------------------------------
Deposit/withdrawal denial       Never silent — enforced
Account freeze / restriction    Never silent — enforced
Payment reversal                Never silent — enforced
Company rejection/revocation    Never silent — add DM + enforce
Loan/card denial (standard)     Never silent — ensure staff cannot bypass
Manual credit during investigation Allow silent — OK
Hold placed (non-restriction)   Allow silent — OK
Scheduled transfer failure      Do not add silent — customer should know
Deal room messages              Never silent — customer must see staff replies

12. RECOMMENDED STANDARDIZED DM TEMPLATES
================================================================================

Short, professional, action-oriented. All include footer "Alta Bank · Newport".

Banking success:
  Title:  {Action} complete
  Body:   {Amount} {direction phrase}. Ref `{code}`.
  Button: View on Alta Bank

Banking failure:
  Title:  {Action} could not be completed
  Body:   We couldn't {action} {amount} {to/from phrase}. {Customer-safe reason}
  Button: Try again

Merchant invoice (recipient):
  Title:  Invoice received
  Body:   {Merchant} sent an invoice for {amount}. Due {date}. Ref `{code}`.
  Buttons: View Invoice · Pay Now

AutoPay:
  Title:  Invoice paid automatically
  Body:   {Merchant} invoice `{code}` for {amount} was paid via AutoPay.
  Button: View invoice

  Title:  AutoPay failed
  Body:   We couldn't auto-pay {merchant}'s invoice. {Reason — customer safe}
  Button: Pay manually

Operator (mandatory):
  Title:  {Customer-facing action title}
  Body:   {Specific account ending {last4} change}.
          Reason: {sanitized operator reason — optional}
  Button: View account

Deal room:
  Title:  Your Secure Deal Room is ready
  Body:   {Welcome one-liner}. Message Alta staff in Discord or on Alta Bank.
  Buttons: Open channel · Open Alta Bank

  Title:  New message from Alta
  Body:   {Preview <=120 chars}
  Button: Open Secure Deal Room

Commercial Pro:
  Title:  Alta Commercial Pro billing failed
  Body:   We couldn't charge {amount} for Pro. Update billing account by {date}.
  Button: Commercial settings

SUMMARY
================================================================================

Alta has a solid notification foundation: ~70 distinct customer DM surfaces,
consistent embed format, retry queue, silent-mode guardrails for compliance
actions, and rich invoice/deal-room buttons where it matters most.

The biggest gaps are incomplete wiring and consistency: company rejection DMs,
payer-side payment failures, AutoPay lifecycle confirmations, payment-engine
pref enforcement, one clear duplicate on recurring invoices, and several
enum-defined types that are never sent.
