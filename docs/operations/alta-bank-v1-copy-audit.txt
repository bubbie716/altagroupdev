Alta Bank V1 — Copy & Microcopy Audit
======================================
Date: 2025-06-28
Scope: Customer-facing Alta Bank + operator /internal console
Method: Codebase review of routes, components, copy modules, empty states, errors, docs
Constraints: Audit only — no UI redesign, no backend changes, no feature work

Overall assessment
------------------
Alta Bank V1 is largely institutional and honest where copy modules exist
(account-status-copy, lending-application-status-copy, secure-deal-room-system-copy,
bank-request-status-copy, ops-copy). The main gaps are **terminology drift** across
surfaces, **preview/V1 language on live paths**, and **generic error fallbacks**
outside mapped domains. Tone is split between premium institutional (lending, deposits)
and promotional marketing (bank dashboard headline, private banking).

Recommended canonical customer vocabulary (V1)
----------------------------------------------
| Concept              | Use everywhere (customer)     | Avoid mixing in              |
|----------------------|-------------------------------|------------------------------|
| Manual review queue  | Waiting on Alta               | Pending Review, Under Review |
| Customer action req. | Waiting on You                | Waiting on you (lowercase)   |
| Lending approved     | Accepted                      | Approved (lending terminal)  |
| Card approved        | Approved                      | Accepted (card terminal)     |
| Deposit/withdraw no  | Denied / not approved         | Declined (activity only OK)  |
| Async comms          | Secure Deal Room              | secure deal room, deal room  |
| Staff                | Alta / Alta Bank              | operator (customer-facing)   |
| RI product name      | Relationship Intelligence     | relationship intelligence    |

Recommended canonical operator vocabulary (V1)
----------------------------------------------
| Action domain        | Primary verb | Status label |
|----------------------|--------------|--------------|
| Bank deposit/withdraw| Approve/Deny | Denied       |
| Lending application  | Accept/Deny  | Accepted/Denied |
| Alta Card application| Approve/Deny | Approved/Denied |
| Company verification | Verify/Reject| Verified/Rejected |
| Thread inbox         | Deal Rooms   | Secure Deal Room (customer-facing ref) |
| Credit ops           | Alta Credit Desk | Credit Desk (nav OK) |

Note: Accept vs Approve is **intentionally split by product** if documented;
UI should not use both verbs for the same action (lending queue says Accept,
dialog says Approve loan application).


1. TERMINOLOGY STANDARDIZED (findings + recommendations)
--------------------------------------------------------

Customer — pending / review (HIGH)
- Requests in Progress: "Waiting on Alta" (bank-request-status-copy.ts) ✓
- Activity ledger: "Deposit · Pending Review" (customer-transaction-copy.ts)
- Account pending badge: "Under Review" vs "Pending Review" (account-status-copy vs account-card.tsx)
- Deposit page: "pending until manually reviewed"
- Success card: "awaiting review"
Fix: Pick ONE customer-facing pending label. Recommend **Waiting on Alta** for
in-flight requests and **Under Review** only for account opening (different object).

Customer — Accepted vs Approved (HIGH)
- Lending filters/status: "Accepted" (lending-application-status-copy.ts)
- Lending definition body: "Your application has been approved." under Accepted key
- System messages: "Your application has been accepted." (secure-deal-room-system-copy.ts)
- Alta Card status: "Approved" / CTA "Accept card"
- Timeline: "Loan accepted" vs "Loan approved"
Fix: Lending terminal = **Accepted** everywhere including system messages.
Card terminal = **Approved**. Do not cross-use.

Customer — Secure Deal Room capitalization (MEDIUM)
- Branded correctly on lending routes and thread views
- Lowercase on some Alta Card surfaces: "Open secure deal room"
Fix: Always **Secure Deal Room** when referring to the product.

Customer — operator vs Alta (LOW)
- withdraw.tsx: "until an operator approves" — replace with Alta-facing language

Operator — Accept vs Approve (HIGH)
- Lending queue: Accept → / workspace button Accept / dialog "Approve loan application"
- Bank queues: Approve
- Alta Card queue: Approve →
Fix: Align button + dialog title per product (Accept for lending, Approve for bank/card).

Operator — Deny vs Reject (MEDIUM)
- Bank: Deny (correct)
- Company verification: Reject (correct)
- Many Deny dialogs say "This will reject the application"
Fix: Dialog body may say reject as verb; keep button **Deny** for bank/lending/card,
**Reject** for company verification only.

Operator — Deal Room naming (MEDIUM)
- Nav: Deal Rooms / page title Deal Room Inbox / empty "secure deal room threads"
- Workspace: Secure Deal Room / Alta Credit Desk
Fix: Nav **Deal Rooms**; customer references **Secure Deal Room**; filter
"Applicant ↔ Alta Credit Desk" is OK as desk label.

Operator — Relationship Intelligence (LOW)
- Nav "Relationships" vs page "Relationship Intelligence" vs lowercase in panels
Fix: Page title **Relationship Intelligence**; nav short label **Relationships** OK;
panel subtitles should capitalize product name.


2. MISLEADING COPY REMOVED (findings — needs copy pass)
-------------------------------------------------------

Live customer paths still using preview / coming soon / V1 disclosure:
- Credit profile tiles: "Coming Soon", "No scores or pre-approvals are calculated in V1"
  (lending-progress.ts, alta-credit-profile-placeholder.tsx)
- Statements: "Generate preview statement" (statement-generate-form.tsx, statements routes)
- Alta Card apply visual: "Preview · terms set at approval"
- Alta Card landing: "no merchant network... in V1"
- Interbank: "Automatic payment execution coming in a future release" (business-future-notice)
- Alta Card RI panel: "Relationship intelligence pending"

Mock-mode only (OK if mock banner visible):
- Bank dashboard "simulated preview data", intrabank "simulated in this preview"
- transfer-form-preview wire simulation strings

Operator visible not-yet-implemented:
- Company RI: "Coming soon" treasury/exchange placeholders
- Pre-approval panel: "Future pre-approval readiness", "Suggested products (placeholder)"
- Compliance: "Full compliance case management is not yet implemented"
- Discord embed: "preview-only until DISCORD_BOT_TOKEN"
- Alta Card workspace: Prisma note exposed to operators

Documentation accurately flags preview areas (metrics-policy.md) but some UI
strings on live bank paths still say "preview" for real features (statements).

Recommendation:
- Statements: drop "preview" — use "Generate statement" if feature is live
- Credit profile: hide section or "Not available yet" without V1 jargon
- Replace "operator" on withdraw page
- Move V1 limitation copy to docs/help, not primary CTAs

6. ERROR MESSAGE IMPROVEMENTS
-----------------------------

Strong pattern (extend)
- formatBankActionError in account-status-copy.ts maps server errors to actionable copy
- Used on deposit/withdraw/transfer forms

Generic fallbacks to replace (customer):
- "Something went wrong while submitting your request…" (bank-request-submission-ui)
- "Request failed" (lending apply)
- "Payment failed" (alta-pay, loan payment, card payment)
- "Application failed" / "Submission failed." / "Generation failed."
- "Unable to complete transfer." / "Upload failed." / "Failed to send."

Generic fallbacks (operator):
- Ad-hoc "Could not …" / "Failed to …" without next step
- ops-confirm-dialog: "This action could not be completed. Try again."

Recommendation: Extend formatBankActionError pattern to payments, lending submit,
and card apply; wire OPS_COPY.actionFailed for operator mutations.


7. SUCCESS MESSAGE IMPROVEMENTS
-------------------------------

Strong (recent work)
- Deposit/Withdraw: "Deposit Submitted" + reassuring body (bank-request-submission-ui)
- No generic "Success!" on bank submit paths

Still generic elsewhere
- Autopay: "Success" / "Failed" status labels
- Some forms rely on green inline text without structured confirmation

Recommendation: Use product-specific success titles everywhere submissions occur
(Alta Pay Sent, Application Submitted, Statement Generated).
