Alta Bank V1 — Performance Hardening Pass
=========================================
Date: 2025-06-25
Scope: Targeted efficiency improvements for several-thousand-user launch.
Constraints: No business logic changes, no UI redesign, no schema changes, no Redis/caching layers.

Build verified: npm run build (pass)


1. DUPLICATE QUERIES ELIMINATED
---------------------------------

Customer bank dashboard (/bank)
- Before: fetchBankDashboardBundle loaded user + accounts, then getUserBankDashboard re-loaded user and fetched accounts again.
- After: getUserBankDashboardBundle loads user once, fetches accounts/transactions in parallel, computes dashboard from in-memory accounts via buildUserBankDashboardMetrics.

Internal ops dashboard (/internal)
- Before: fetchEnhancedDashboard called getInternalDashboardMetrics() twice (once directly, once inside getOpsHealth()).
- After: Metrics fetched once; buildOpsHealthFromMetrics(metrics, maintenance) reuses preloaded counts.
- negativeBalances and largeAdjustmentsLast30Days folded into getInternalDashboardMetrics (removed duplicate dashboard queries).

Internal bank ops / queue routes
- Before: All queue pages and bank index used fetchInternalBankOps (summary + all accounts + all pending queues).
- After:
  - /internal/bank → fetchInternalBankOpsSummary only
  - /internal/queues/deposits → fetchPendingDepositsQueue only
  - /internal/queues/withdrawals → fetchPendingWithdrawalsQueue only
  - /internal/queues/account-openings → fetchPendingAccountOpeningsQueue only
  - fetchInternalBankOps retained for backward compatibility

Global search
- Before: Separate Prisma queries for deposit-type and withdrawal-type transactions in addition to general transaction search.
- After: Single bankTransaction.findMany; deposit/withdrawal/alta_pay results classified from shared result set.

Customer 360 workspace
- Removed unused Alta Pay transaction queries from getInternalCustomer360 (returns empty altaPayActivity array; UI unchanged).

Company 360 workspace
- Removed unused statements list and audit count queries from loader (statements returns []; audit loaded only on audit tab).

Historical reports (ops-reports.service)
- Merged deposit/withdrawal aggregates into one groupBy (reviewedAt filter preserved).
- Transfer count derived from findMany length (removed separate count query).
- ADJUSTMENT metrics kept as separate aggregate (createdAt filter — business rule preserved).


2. DUPLICATE CALCULATIONS REMOVED
------------------------------------

Customer dashboard balance buckets
- buildUserBankDashboardMetrics: single pass over accounts for checking/savings/private/money-market/business/total (was multiple filter+reduce passes).

Relationship Intelligence cron
- refreshAllRelationshipProfiles now passes skipRecommendations: true to avoid duplicating work already done by the recommendations cron job.

Internal dashboard health
- Platform health detail built from preloaded metrics instead of re-querying account/user counts.


3. PRISMA IMPROVEMENTS
----------------------

Relationship timeline dedupe (relationship-timeline.service)
- Before: hasDedupeKey / findTimelineEventByDedupeKey loaded all timeline events for a user and scanned in memory.
- After: findFirst with JSON-path filter on metadata.dedupeKey — O(1) indexed lookup per check.

Bank statement scheduler (bank-statement-scheduler.service)
- Before: Per inactive account, separate count query for period activity (N+1).
- After: Single groupBy on bankAccountId for all inactive accounts in one query.

Alta Card autopay cron (alta-card-autopay.service)
- Before: Per-card findFirst for due statement.
- After: Batch findMany for all autopay-eligible cards; Map lookup per card.

Reports groupBy
- Reduced parallel query count for transaction volume metrics while preserving date-field semantics per type.


4. REACT / RENDER IMPROVEMENTS
------------------------------

Global search (internal-global-search.tsx)
- 300ms debounce on keystrokes — reduces server round-trips during typing.
- Request ID guard — stale in-flight responses discarded when query changes.

No broad memoization pass applied (no measured re-render hotspots at V1 scale).


5. WORKSPACE LOADING IMPROVEMENTS
-----------------------------------

Customer workspace (/internal/users/$userId)
- Universal timeline (buildUniversalCustomerTimeline, ~60 events) loaded only when tab=activity.
- Overview/accounts/relationship tabs skip timeline query.

Company workspace (/internal/companies/$companyId)
- Universal timeline loaded only when tab=activity.
- Audit logs loaded only when tab=audit.
- Relationship recommendations + dedicated timeline loaded only when tab=relationship.
- Relationship sidebar detail still always loaded (required for overview sidebar).


6. TIMELINE OPTIMIZATIONS
---------------------------

- Tab-conditional universal timeline loading (customer + company workspaces).
- Timeline dedupe/backfill: indexed findFirst instead of full-table scan per dedupe key.


7. SEARCH OPTIMIZATIONS
-----------------------

- 2 fewer DB queries per search (deposit + withdrawal dedicated queries removed).
- Client debounce + stale-response guard.
- Search coverage unchanged — all entity types still queried in parallel.


8. DASHBOARD OPTIMIZATIONS
----------------------------

Customer /bank
- ~2 fewer account list queries per page load.
- ~1 fewer user load per page load.
- Balance aggregation in single CPU pass.

Internal /internal
- ~18 fewer Prisma round-trips on dashboard load (eliminated duplicate metrics fetch).
- Queue aging still loaded separately (see V2).

Internal bank index + queues
- Queue pages load only their queue data (~3–4 fewer heavy list queries per queue page visit).


9. ESTIMATED PERFORMANCE IMPROVEMENTS
---------------------------------------

Qualitative estimates at several-thousand-user scale (no formal benchmarks run):

| Surface                         | Before (approx)     | After (approx)      | Notes                          |
|---------------------------------|---------------------|---------------------|--------------------------------|
| Customer /bank load             | 4–5 account-related | 2–3 account-related | User + accounts deduped        |
| Internal dashboard              | ~36 DB ops          | ~18 DB ops          | Metrics dedupe                 |
| Queue page (deposits/etc.)      | Full bank ops bundle| Single queue list   | Largest win on queue workflows |
| Global search keystroke         | 1 req/char          | 1 req/300ms         | ~70% fewer requests while typing |
| Global search server            | +2 tx queries       | Baseline            | Per completed search           |
| RI profile refresh cron         | 2× rec per user     | 1× rec per user     | skipRecommendations              |
| Statement scheduler (inactive)  | O(N) counts         | O(1) groupBy        | N = inactive accounts          |
| Alta Card autopay cron          | O(cards) findFirst  | O(1) batch + loop   | cards = autopay-enabled        |
| Company workspace (overview)    | timeline+audit+RI   | RI sidebar only     | Tab-conditional                |
| Timeline dedupe/backfill        | O(events) scan      | O(1) lookup         | Per dedupe check                 |

Expected user-visible impact:
- Faster internal dashboard and queue pages under operator load.
- Snappier /bank dashboard.
- Reduced DB connection pressure during search and cron windows.
- Lower latency opening company/customer workspace on non-activity tabs.


10. REMAINING FUTURE OPTIMIZATIONS (V2 ONLY)
--------------------------------------------

Deferred — not worth complexity at V1 scale or require measurement first:

- Loan servicing auto-pay: earlierUnpaid findFirst per schedule item (loan.service.ts) — batch unpaid lookup.
- Scheduled transfer executor: per-payment query burst during execution runs.
- getCustomerRelationshipView: potential triple profile calculation on customer-facing RI pages.
- Queue aging metrics: overlap with dashboard counts — could merge with metrics bundle.
- Customer workspace audit tab: recentAuditLogs still loaded via getInternalUserDetail on every tab.
- Company statements tab: dedicated lazy loader if statements UI is re-enabled.
- OpsTable pages: true server-side pagination/filtering (current hard caps remain).
- Milestone reconciliation: single-pass transaction replay optimization.
- React memoization: targeted only after profiler identifies hot components.
- Redis / background workers / response caching: not needed at V1 scale.
