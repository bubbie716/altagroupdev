ALTA CARD PRODUCTION READINESS AUDIT
======================================
Date: 2025-06-25
Scope: End-to-end audit (18 dimensions). Audit only — no implementation.

VERDICT
-------
Alta Card is a strong V1 intrabank revolving credit product with real billing, interest,
fees, autopay, Alta Pay funding, business/employee lines, relationship pricing, and
extensive audit logging. It is NOT production-ready as a regulated consumer/business
credit card without closing critical gaps in admin controls, customer communications,
statement delivery, testing, attachment security, and card-network semantics.

The codebase is unusually well-documented (docs/banking/alta-card.md, ~800 lines) and
architecturally coherent for a simulated bank. The remaining work is mostly operational
hardening and product completeness, not greenfield implementation.

================================================================================
CRITICAL ISSUES
================================================================================

C1 — No card processor / authorization model
Severity: Critical
Description: PURCHASE transaction type exists but there is no authorization/capture
flow, merchant network, or real-time decline semantics. Spending is simulated via
admin adjustments, cash advance, Alta Pay charges, and manual ops—not POS/card-present
behavior.
Why it matters: A "production financial product" labeled Alta Card implies spend
controls, merchant categories, holds, and dispute workflows. Without them, the product
is intrabank credit simulation only.
Recommended implementation: Define V1 scope explicitly (simulation vs network card).
If simulation-only, remove/rename PURCHASE from user-facing copy and document supported
spend channels. For production card behavior, integrate authorization service with
hold/release, MCC rules, and dispute case management.

C2 — Dual admin API paths with weaker validation on list panel
Severity: Critical
Description: Internal list view (internal-alta-card-panel.tsx) calls
updateAltaCardLimitRecord, updateAltaCardRateRecord, etc. from alta-card.service.ts —
no required reason, limit can be set below balance (available credit clamped to 0).
Detail ops panel uses alta-card-admin.service.ts with required reason, balance
validation, and admin override for edge cases.
Why it matters: Operators can change credit terms without audit reason or balance
checks from the main card list—violates credit ops controls and can create negative
available credit / incoherent utilization.
Recommended implementation: Deprecate admin mutations in alta-card.service.ts. Route
all internal changes through alta-card-admin.service.ts + alta-card-admin.functions.ts.
Single UI entry point: detail ops panel only.

C3 — No automated test suite for billing/financial logic
Severity: Critical
Description: Docs explicitly state manual billing checklist only. No *.test.* /
*.spec.* files cover Alta Card payment allocation, interest accrual, statement close,
late fees, or autopay idempotency.
Why it matters: Billing regressions are silent and expensive. Minimum payment rules,
overdue interest, and payment waterfall are exactly the logic that needs regression
protection in a financial product.
Recommended implementation: Unit tests for alta-card-payment-allocation.ts,
alta-card-minimum-payment.ts, alta-card-billing-cycle.ts. Integration tests for
statement close → overdue → interest → late fee → autopay success/failure paths
using fixed clocks.

C4 — No customer notifications (autopay, due reminders)
Severity: Critical
Description: Documented TODOs in alta-card-autopay.service.ts and
docs/banking/alta-card.md: autopay success, autopay failed, payment due reminder —
not implemented.
Why it matters: Customers miss due dates; failed autopay runs silently. Real banks
treat these as compliance and delinquency-prevention requirements.
Recommended implementation: Hook into existing notification infrastructure
(Discord/email/in-app). Trigger on ALTA_CARD_AUTOPAY_SUCCESS / FAILED, and scheduled
job N days before dueDate for open statements.

C5 — Application/review thread attachments use public blob access
Severity: Critical
Description: Upload service sets access: "public" on Vercel Blob for Alta Card
application thread files.
Why it matters: Financial KYC/support documents (IDs, income proof) must not be
world-readable URLs. High severity data exposure risk.
Recommended implementation: Private blobs with signed URLs, auth-gated download API,
virus scan hook if available. Audit every download.


================================================================================
HIGH ISSUES
================================================================================

H1 — DELINQUENT card status never set automatically
Severity: High
Description: Statements transition to OVERDUE via markOverdueStatements(), but card
status is not promoted to DELINQUENT. Status allows payments/autopay on DELINQUENT but
nothing assigns it.
Why it matters: Delinquency workflows (collections, freeze escalation, rate changes)
depend on card-level status. Operators must manually intervene.
Recommended implementation: After overdue + grace period (or immediately per policy),
update card status to DELINQUENT, audit ALTA_CARD_STATUS_CHANGED, optionally block new
spend.

H2 — EXPIRED status unused in automation
Severity: High
Description: Enum blocks transactions but no cron or lifecycle sets cards to EXPIRED
(e.g., closed applications, term expiry, inactive reissue).
Why it matters: Dead enum values confuse ops and allow inconsistent manual-only
transitions.
Recommended implementation: Define expiry rules (e.g., PENDING > 90 days → EXPIRED)
and document manual EXPIRED use cases.

H3 — GENERATED statement status creates operator confusion
Severity: High
Description: Users/admins can create preview/generate statements outside the OPEN →
ISSUED month-end close flow. GENERATED appears in status taxonomy but is not clearly
distinguished from production-issued statements in all UIs.
Why it matters: Statement lists may clutter with non-official periods; interest/billing
may not align with generated previews.
Recommended implementation: Restrict GENERATED to admin-only preview tool, or remove
from user-visible lists. Document lifecycle vs production ISSUED.

H4 — Business card duplicate ledger reconciliation
Severity: High
Description: Application layer blocks second business card per company, but runtime
paths include reconcileBusinessCardLedger and hasMultipleBusinessCards checks—
suggesting duplicates can exist from legacy/data issues.
Why it matters: Business line balances and employee rollup depend on single parent card
invariant.
Recommended implementation: DB unique constraint on (companyId) for active business
cards where possible. Reconciliation job that flags/alerts duplicates. Admin tool to
merge or close extras.

H5 — No dispute / chargeback workflow
Severity: High
Description: No dispute case model, provisional credit, or merchant reversal flow for
card transactions.
Why it matters: Expected for any card product handling customer spend (including Alta
Pay charges).
Recommended implementation: V2: AltaCardDispute model, statuses, admin queue, link to
original transaction, audit trail.

H6 — Cron system actor falls back to first ADMIN user
Severity: High
Description: Scheduled jobs use a system user derived from first admin when no dedicated
service account exists.
Why it matters: Audit attribution is wrong; job fails entirely if no admin user exists.
Recommended implementation: Dedicated SYSTEM or CRON service user in schema. Never
depend on human admin roster.

H7 — Staff assignment on Alta Card threads (inconsistent with lending)
Severity: High
Description: Alta Card application and review threads still support
assignAltaCardThreadStaff / assignAltaCardReviewThreadStaff with UI on internal review
pages. Lending recently deprecated individual banker assignment.
Why it matters: Inconsistent ops model across products; may reintroduce "assigned
banker" UX removed from lending.
Recommended implementation: Align policy: either remove assignment from Alta Card
threads (mirror lending) or document why review assignment remains for credit ops.

H8 — Documentation currency inconsistency
Severity: High
Description: Tier default limits in docs/tables sometimes use $; platform currency is
Florins (formatAltaCardCurrency → ƒ).
Why it matters: Operator and customer confusion in a single-currency simulated economy.
Recommended implementation: Global find/replace in docs and tier config comments to
ƒ. Single currency formatter for docs generation if needed.


================================================================================
MEDIUM ISSUES
================================================================================

M1 — Two internal admin panels with overlapping scope
Severity: Medium
Description: internal-alta-card-panel.tsx (list + quick mutations) vs
internal-alta-card-ops-panel.tsx (full ops on detail route). Different capabilities
and validation rules.
Why it matters: Operators use the wrong panel; weaker path becomes default.
Recommended implementation: List panel read-only + links to detail ops. Consolidate
mutations in ops panel only.

M2 — Duplicate thread service architecture (~400 lines each)
Severity: Medium
Description: Parallel implementations for application threads vs review threads
(upload, assign staff, messaging, audit).
Why it matters: Bug fixes must be applied twice; behavior drifts.
Recommended implementation: Extract shared secure-thread.service with product-
specific metadata adapters.

M3 — Limit/rate/tier/status mutations duplicated across services
Severity: Medium
Description: alta-card.service.ts and alta-card-admin.service.ts both expose limit,
rate, tier, freeze, close, adjustments with different rules.
Why it matters: See C2; architectural debt.
Recommended implementation: Single write model in admin service; user-facing service
read-only + user actions (activate, autopay, pay).

M4 — Alta Pay volume analytics TODO
Severity: Medium
Description: Docs note centralized Alta Pay volume analytics not wired for
relationship pricing factors.
Why it matters: Relationship pricing underweights Alta Pay spend history.
Recommended implementation: Add alta_pay_volume factor when analytics service exists.

M5 — Cash advance without ATM/network semantics
Severity: Medium
Description: Cash advance posts as internal transaction type; no ATM limit, daily cap,
or fee structure beyond generic charges.
Why it matters: Real cards separate purchase APR vs cash APR and cash advance fees.
Recommended implementation: Document V1 as "internal cash draw" or add cash-specific
fee + rate rules.

M6 — Employee card model vs unused EMPLOYEE card type enum
Severity: Medium
Description: Employee cards use AltaEmployeeCard model rolling to business parent, not
AltaCardType.EMPLOYEE.
Why it matters: Schema/API confusion for future features.
Recommended implementation: Remove or document deprecated enum value; ensure all
queries use employee card model consistently.

M7 — Manual billing test checklist only in docs
Severity: Medium
Description: Comprehensive manual checklist exists but no CI enforcement.
Why it matters: Checklist rot; human error on billing changes.
Recommended implementation: Encode checklist as automated tests (see C3).

M8 — Cron deployment documented as TODO
Severity: Medium
Description: vercel.json cron example marked deployment TODO; production relies on
/api/cron/scheduled-transfers being triggered externally.
Why it matters: Missing cron = no statements, interest, or autopay in production.
Recommended implementation: Commit production cron config, monitor OpsJobRun failures,
alert on missed runs.

M9 — Review eligibility / upgrade path UX partially hidden
Severity: Medium
Description: Review form and BankReviewButton exist; eligibility rules and cooldowns
may not be obvious on main dashboard.
Why it matters: Private clients may not discover limit/rate review workflow.
Recommended implementation: Persistent dashboard CTA when eligible; copy aligned with
relationship pricing factors.


================================================================================
LOW ISSUES
================================================================================

L1 — Random cardLastFour on issue
Severity: Low
Description: Last four digits generated at card creation, not tied to a PAN (expected
for simulation).
Why it matters: Cosmetic only unless marketed as real card numbers.
Recommended implementation: Document as display-only; optional user-chosen alias.

L2 — LOST status allows some operations
Severity: Low
Description: LOST in active statuses for some paths; behavior vs FROZEN not always
obvious in UI.
Why it matters: Ops may use wrong status for stolen card scenario.
Recommended implementation: Status transition matrix in docs + admin UI tooltips.

L3 — Interest waiver via adjustment credits only
Severity: Low
Description: Docs say use adjustment credits to waive interest; no dedicated "waive
interest" admin action.
Why it matters: Works but non-obvious for operators.
Recommended implementation: Admin "waive statement interest" with reason, tied to
statement row.

L4 — OpsJobRun visibility only on internal alta-card page
Severity: Low
Description: Job history not centralized in platform ops dashboard.
Why it matters: Harder to correlate with other bank cron failures.
Recommended implementation: Link from /internal hub or maintenance panel.

L5 — Server-side PDF stub is dead code (client PDF works)
Severity: Low
Description: CORRECTION (2025-06-25): Alta Card statement PDF export IS implemented.
AltaCardStatementDocument (src/components/bank/alta-card/alta-card-statement-document.tsx)
provides a working "Download PDF" button on statement detail routes, using the same
client-side html2canvas + jsPDF path as bank account statements
(src/lib/bank/download-statement-pdf.ts). Print is also supported.
What is NOT implemented: server-side generateStatementPdf() in
src/server/alta-card-statement-pdf.ts and downloadAltaCardStatementPdfRecord — these
return "not yet available" and are not wired to any UI.
Why it matters: The audit originally flagged this as Critical incorrectly. docs/banking/
alta-card.md still says "PDF (placeholder)" which is misleading. Dead server stub adds
confusion.
Recommended implementation: Update alta-card.md to document client-side PDF export.
Remove or implement the unused server fn. Optional V2: server-rendered PDFs for email
delivery and archival (not required for current UX).

================================================================================
PRODUCTION READINESS CONCLUSION
================================================================================

For intrabank revolving credit inside Alta Bank (Alta Pay funding, manual/cash-advance
spend, automated billing, relationship-priced limits):
  NEAR production-ready after P0 (especially admin consolidation, attachment security,
  tests, notifications).

For a regulated consumer/business credit card product:
  NOT production-ready. Critical gaps in network semantics, disputes, regulatory
  disclosures, statement email delivery, and customer communications must be closed OR
  scope must be explicitly bounded to "simulated credit line" in all customer-facing
  materials.

Recommended next step: Treat P0 as a launch gate checklist; assign owners per item;
do not enable maintenance-mode production cutover for Alta Card until C2, C5, C3, and
C4 are resolved.
